Section 1

Executive Summary

Gaurav Nandhan BA Studio is an AI-powered, enterprise-grade Business Analysis platform built on Next.js 15 and deployed on Vercel. It transforms raw requirements — supplied as Minutes of Meeting (MOM), voice recordings, or document uploads — into a full suite of BABOK v3-compliant professional artifacts.

The platform eliminates the hours-long manual effort traditionally required to produce BRDs, FRDs, PRDs, SRDs, UML diagrams, wireframes, prototypes, test cases, and executive pitches. Beyond document generation, it provides a living Intelligence Panel with continuous feasibility analysis, regulatory scanning, conflict detection, gap analysis, and strategic opportunity identification.

✅
Core Value Proposition
A BA analyst that previously spent 3–5 days producing a complete set of project artifacts can now generate the same output in under 30 minutes, with AI confidence scores and human-review flags ensuring quality assurance before sign-off.

Section 2

Product Overview

2.1 Problem Statement

Business analysts in enterprise environments spend a disproportionate amount of time on documentation tasks that are repetitive, error-prone, and siloed. Common pain points include:

  • Manually converting unstructured meeting notes into structured BRDs and FRDs
  • Producing consistent UML diagrams and flowcharts without specialized tooling
  • No automated detection of contradictions, regulatory flags, or logic gaps in requirements
  • Inability to rapidly prototype UI flows for stakeholder review
  • No real-time impact analysis when requirements change mid-project
  • Lack of integrated traceability from requirements through test cases

2.2 Solution

BA Studio provides an integrated workspace where a Business Analyst pastes MOM, requirements, or audio recordings into a chat interface. An AI agent processes the input through multiple analytical lenses and generates all required artifacts on demand. A real-time Intelligence Panel continuously monitors the session for quality, completeness, and risks.

Section 6

Feature Requirements — Core BA Workflow

6.1 AI-Powered Chat Interface

The primary interaction surface. A full-screen chat panel allows the BA to have a natural conversation with the AI agent about project requirements.

Input Methods

  • Text Input: Multi-line auto-expanding textarea. Supports Enter to send, Shift+Enter for newline.
  • File Attachment (📎): Accepts .txt, .md, .csv files. Content is appended to the chat input with filename prefix.
  • Audio Upload (🎤): Accepts audio/* MIME types. Audio is base64-encoded and sent to the Gemini API for transcription + analysis. BA does not need to transcribe manually.
  • Paste: Native clipboard paste; full MOM text can be pasted directly.

Streaming Response

AI responses stream token-by-token via the ReadableStream API. UI updates are throttled to every 100ms to prevent browser hang during long document generation. A "typing cursor" animation (animate-pulse) is displayed during streaming. Framer Motion animations (blur fade-in) are applied to each new message bubble.

Message Rendering

  • All messages are rendered with react-markdown and remark-gfm, supporting tables, code blocks, bold, italic, and lists.
  • Assistant messages: blue-gradient Brain avatar, dark card background.
  • User messages: right-aligned, blue gradient card.

Session State Machine

The chat has a three-state lifecycle driven by the Analyze API:

StateDescriptionDocuments Available?
INTAKEInitial state — waiting for first MOM inputNo
QUESTIONINGAI has detected requirements; asking clarifying questionsYes (partial)
READYAll 7 readiness checklist items passed; full artifact suite unlockedYes (full suite)

AI Confidence System

Every generated document includes a trailing metadata tag that the system parses:

[CONFIDENCE: 87% | REVIEW: REQUIRED | LINKS: REQ-1,REQ-3 | REASON: Missing SLA definitions]
  • Confidence %: Displayed as a color-coded badge (green ≥90%, amber ≥70%, red <70%)
  • Review flag: REQUIRED (⚠️) or Verified (✅)
  • Links: Requirement IDs this document is derived from — used for stale detection
  • Reason: Human-readable explanation of the confidence score

Stale Document Detection

When requirements change after document generation, affected documents are automatically flagged as stale via the Semantic Graph BFS traversal. A visual warning indicator appears on stale sidebar items, and a "Regenerate Stale Artifacts" button appears in the Cascading Impact panel.

Regenerate with Updates

Each document toolbar shows a "Regenerate with vN Updates" button when the scope history has more than one version. This re-generates the document with the latest requirement snapshot, ensuring documents reflect current state.

Section 6 (continued)

Diagram Generation & Visual Outputs

6.3 UML Diagram Generation (PlantUML)

The AI generates valid PlantUML source code wrapped in @startuml ... @enduml tags. The system:

  1. Extracts PlantUML code from the AI response (regex match or JSON .code field)
  2. Applies cleanup passes: strips !theme directives, removes markdown artifacts (**, \_)
  3. Encodes the source using pako deflate → custom PlantUML base64 alphabet
  4. Constructs a https://www.plantuml.com/plantuml/svg/~1{encoded} URL
  5. Renders the SVG as an <img> tag with loading state and error fallback

A collapsible "PlantUML Source" section and an "Open & Edit in PlantUML Editor" external link are always shown below the rendered diagram.

6.4 Mermaid Flowchart Generation

Flowcharts are rendered entirely in the browser using the Mermaid.js library. The system applies a multi-pass scrubber to handle AI-generated code that may contain Mermaid-incompatible syntax:

  • Strips all parentheses (which break Mermaid node labels)
  • Normalizes arrow syntax to -->
  • Wraps subgraph titles containing special characters in double quotes
  • Forces graph TD header if no valid diagram type is detected
  • 1-second debounce applied during streaming to avoid partial-render errors

On render error, a "Visual Engine Fallback" panel displays the raw Mermaid code alongside a human-readable error so the user can still access the logic.

6.5 Wireframe Builder (DynamicUIBuilder)

Wireframes are not static images — they are generated as a JSON component schema and rendered dynamically by the DynamicUIBuilder component. The schema describes a hierarchy of UI components (headings, cards, inputs, buttons, navigation bars, data tables) which are rendered as styled React elements.

ℹ️
Design Rationale
Using a JSON schema instead of static HTML means wireframes are portable, editable, and semantically meaningful. Each component in the schema can be individually analyzed for accessibility and design consistency.

If the AI returns a JSON object with a .code string field, that string is used as the schema. Fallback parsing extracts the first JSON object from the raw content. A blank-state message prompts the user to click "Edit" if the schema cannot be parsed.

6.6 Interactive Prototype Preview

Prototypes are generated as full HTML documents with embedded Tailwind CSS and Alpine.js. They are rendered in a sandboxed <iframe> with the following security constraints:

  • sandbox="allow-scripts allow-modals allow-forms allow-same-origin allow-popups"
  • External navigation links (<a href="...">) are intercepted and cancelled unless they point to #
  • Form submissions are prevented (evt.preventDefault())

Tailwind CDN and Alpine.js CDN are auto-injected if not already present in the generated HTML. A 2-second debounce during streaming prevents partial HTML from rendering in the iframe. A language selector dropdown (English, Telugu, Hindi, Arabic) is shown in the header when the Prototypes tab is active.

Section 7

Intelligence Panel Features

The Intelligence Panel is a persistent right-hand sidebar that runs continuously alongside every chat session. It is populated by the /api/analyze endpoint which is invoked in parallel with every user message. The panel contains 7+ distinct analytical modules:

7.1 Feasibility & SME Readiness Analyzer

The readiness score is a 0–7 scale tracked against a structured checklist. Each checklist item maps to a fundamental BA completeness criterion:

#Checklist ItemDescription
1hasProblemStatementClear problem or opportunity defined
2hasStakeholdersKey stakeholders identified
3hasScopeIn-scope and out-of-scope boundaries defined
4hasFunctionalReqsAt least 3 functional requirements articulated
5hasNonFunctionalReqsPerformance, scalability, or security constraints mentioned
6hasTimelineDelivery timeline or phase structure indicated
7hasTechContextTechnology stack or integration constraints mentioned

The UI displays a colored progress bar (red → amber → green) and individual dot indicators for each criterion. The session transitions to READY state when all 7 items pass, unlocking the complete document generation suite.

Feasibility Issues list structural gaps detected in the requirements. Contradictions surface internal inconsistencies. Both are displayed as expandable lists in the Intelligence Panel.

7.2 Conflict Detector

The AI identifies conflicts between different stakeholder requirements or between different parts of the same MOM. For each conflict detected, the system shows:

  • A plain-English description of the conflicting statements
  • A proposed resolution with a recommended compromise approach

Conflicts are highlighted in a red-tinted card in the Intelligence Panel and must be manually reviewed and resolved by the BA before proceeding.

7.3 Regulatory Advisor

Automatically detects domain-specific regulatory obligations from the project domain (e.g., Healthcare → HIPAA; FinTech → PCI-DSS, RBI guidelines; EdTech → FERPA; E-Commerce → GDPR). Flags are displayed as bullet points with an "Add to Scope" button that appends the regulatory requirement to the chat input for the AI to incorporate.

7.4 Logic Debugger

Detects logical inconsistencies and process design flaws that are not immediately visible as text contradictions. Each alert contains:

  • Type: Category of logic issue (e.g., "Circular Dependency", "Orphaned Process", "Undefined Exception Handler")
  • Description: Precise description of the issue
  • Risk: Business impact classification (HIGH / MEDIUM / LOW)

7.5 Requirement Gap Analysis

Identifies categories of requirements that are entirely missing from the MOM. Each gap is classified by:

  • Area: The missing domain (e.g., "Error Handling", "Reporting", "Notification System")
  • Severity: HIGH or MEDIUM
  • Gap description: Specific missing requirement

An "Auto-Fill Gap Prompts" button appends all identified gaps as bullet points to the input field, allowing the BA to quickly prompt the AI to address them.

7.6 Billion Dollar Opportunities Engine

A forward-looking module that analyzes the project context and suggests adjacent market opportunities, potential pivot points, or feature extensions that could dramatically increase the business value of the product. Each opportunity is shown with:

  • Title: The opportunity name
  • Impact: Description of business impact
  • Effort: Low / Medium / High effort estimate (color-coded badge)

7.7 Strategic Moats Advisor

Identifies competitive moats (network effects, data advantages, switching costs, proprietary technology) present or achievable in the project context. Displayed as a list of strategic assets the product can build over time.

7.8 SME Domain Insight

Provides a subject-matter expert perspective on the detected domain. For example, for a banking project, the SME Insight might surface industry-specific concerns like Core Banking System (CBS) integration patterns, SWIFT messaging compliance, or real-time settlement requirements. An "Explore deeper" link appends the insight as a prompt for further AI analysis.

7.9 Impact Score

After scope changes, a numeric impact score (0–10) is computed and displayed alongside a summary of what changed. This score drives the cascading impact detection and stale document identification.

Section 8

Semantic Knowledge Graph Engine

The Semantic Graph Engine is the most technically sophisticated component of BA Studio. It maintains a directed property graph of all project artifacts and their relationships, enabling precise downstream impact analysis with zero additional API cost.

8.1 Node Types

Node TypeExample IDsDescription
REQUIREMENTREQ-1, REQ-2Individual functional or non-functional requirements
EPICEPIC-1Large feature groupings (SAFe/Agile)
FEATUREFEAT-01Mid-level feature decomposition
SCREENSCR-01, SCR-04UI screens or pages
APIAPI-01, API-03API endpoints or services
TEST_CASETC-01, TC-12Individual acceptance test cases
DOCUMENTBRD, FRDGenerated BA artifact documents

8.2 Relationship Types (Edges)

RelationshipDirectionMeaning
CONTAINSEpic → Feature → RequirementHierarchical decomposition
RENDERS_ONRequirement → ScreenThis requirement is realized on this UI screen
CALLSRequirement/Feature → APIThis requirement triggers this API call
VERIFIED_BYRequirement → Test CaseThis requirement is validated by this test
DOCUMENTED_INRequirement/Feature → DocumentThis requirement is documented in this artifact

8.3 BFS Traversal Algorithm

When requirements change (added, modified, or removed), the engine runs a breadth-first search (BFS) from the changed node IDs to find all transitively affected downstream artifacts:

  1. Build an adjacency map in memory using buildInMemoryGraph(nodes, edges)
  2. Seed the BFS queue with all changed requirement node IDs
  3. Traverse the graph, collecting all reachable nodes and recording traversal paths
  4. Map each affected node type to its downstream BA document types (e.g., SCREEN → Wireframes, Prototypes)
  5. Flag only the precisely affected document types as stale

8.4 Graph Persistence

Graph data is persisted to PostgreSQL via upsertGraph() called server-side after each analyze response. The function is fire-and-forget — it never throws or blocks the main flow. The database tables are GraphNode and GraphEdge, scoped to a project.

8.5 Knowledge Graph Panel UI

The right-hand panel displays a live count of nodes by type, the number of relationships mapped, the last BFS traversal paths (first 3 shown inline), and a "View Full Impact ↗" button that opens a modal with the complete traversal result grouped by node type.

📌
Design Principle
The graph traversal is a pure JavaScript function that runs entirely in the browser with zero API calls. This means impact analysis is instantaneous, costs nothing per query, and works for all users including unauthenticated guest users.
Section 9

Scope Evolution & Versioning Engine

9.1 Requirement Snapshots

Every analyze API response returns a structured snapshot — an array of RequirementNode objects representing the current understanding of all project requirements. Each node has an id (e.g., REQ-1), a text description, and a status (CONFIRMED | PROPOSED).

9.2 Delta Calculation (calculateDelta)

When a new snapshot is received, the calculateDelta(baseline, current) function computes the structural diff:

  • Added: Requirements in current but not in baseline
  • Modified: Requirements where the text or status changed
  • Removed: Requirements in baseline but absent from current
  • Impact Score: min(10, ceil((changeCount / baselineLength) × 10))
  • Architectural Conflict: Raised when the detected domain shifts mid-session

9.3 Scope History

Each snapshot and its delta report are stored in the scopeHistory state array. The version number visible in the document toolbar ("Version 3.0 (Revised)") corresponds to the length of this array.

9.4 Scope Evolution Timeline Modal

Accessible from the "Requirement Timeline" button in the sidebar. Displays a vertical chronological timeline of all scope versions, each with:

  • Version number and timestamp
  • Animated timeline dot (pulsing green for the latest version)
  • Full delta report: ADDED (green), MODIFIED (amber), REMOVED (red) requirement cards
  • Impact Score badge (green ≤4, amber 5–7, red >7)
  • Architectural conflict warning if domain shifted
  • Semantic Graph traversal paths for the latest version
Section 10

Traceability Matrix

The Traceability Matrix is a dedicated view in the sidebar workspace (accessible via the Network icon). It provides a tabular cross-reference of all project artifacts showing which requirements map to which documents, screens, APIs, and test cases.

10.1 Data Source

The Traceability Matrix is populated from the in-memory Semantic Graph (graphNodes and graphEdges state). Each row in the matrix represents a requirement node, with columns for related Screens, APIs, Test Cases, and Documents derived from the outgoing edges.

10.2 Requirements

  • Must display all REQUIREMENT and FEATURE nodes in the graph
  • Each row shows: Requirement ID, Label, linked Screens, linked APIs, linked Test Cases, linked Documents
  • Empty cells displayed as "—" to clearly indicate unmapped artifacts
  • Table must be printable as part of the PDF export

Export & Share Features

11.1 Single Document PDF Export

The "PDF Export" button in the document toolbar opens a new browser window with a print-ready HTML document and triggers the native browser print dialog (window.print()). The user saves as PDF via the browser's PDF printer. Features:

  • Professional header with document name, generation timestamp, and BABOK badge
  • AI Confidence badge with color coding in the header
  • Full markdown-to-HTML conversion supporting: headings, paragraphs, tables, lists, code blocks
  • Alternate row shading on tables for readability
  • Professional footer with branding and "Confidential & Proprietary"
  • For UML Diagrams: the rendered SVG image is captured and embedded in the PDF
  • For Wireframes/Prototypes: raw HTML source is shown with a visual note explaining the interactive content

11.2 Export All to PDF (Bulk Export)

The "Export All (PDF)" button in the main header generates a single comprehensive report containing all generated documents. The bulk report includes:

  • Cover Page: "Gaurav Nandhan BA Studio — Full Business Analysis Report" with date and BABOK badge
  • Table of Contents: Numbered list of all included documents
  • Document sections: Each generated document on a new page with its confidence badge

11.3 Download (Raw File Export)

The "Export" button downloads the document as a raw file:

  • Text documents (BRD, FRD, PRD, etc.): Downloaded as .md (Markdown)
  • Wireframes and Prototypes: Downloaded as .html

11.4 Copy to Clipboard

The "Copy" button copies the raw document content to the system clipboard using the Clipboard API. A checkmark animation confirms the copy action (2-second auto-reset).

11.5 Public Share Links

After saving a session, a shareable URL is generated in the format /share/{projectId}. Anyone with the link can view the project's chat history and generated documents without authentication. The share page is served from /src/app/share/[id] as a read-only view. The link is copied to clipboard via the "Share Link" button.

⚠️
Security Note
Share links are public by URL — anyone with the link can view the project. Users should be aware of this before sharing sensitive requirement documents. Future v2 will support password-protected shares.
Section 12

Jira Integration

The Jira Sync feature allows BAs to push generated requirements from FRD documents directly into Jira as user stories or epics without leaving the BA Studio interface.

12.1 JiraModal Component

Triggered by the "Jira Sync" button in the document header. The modal is available from both the main header toolbar and the individual document toolbar. The JiraModal component (src/components/JiraModal.tsx) receives the document title and content as props.

12.2 Configuration

The modal requires the user to provide:

  • Jira instance base URL (e.g., https://company.atlassian.net)
  • Jira API Token (user-provided; not stored server-side)
  • Jira Project Key (e.g., PROJ)
  • Issue type (Story / Epic / Task)

12.3 Sync Behavior

The system parses the FRD document content to extract individual requirements, formats them as Jira issue descriptions, and creates issues via the Jira REST API (routed through /api/integrate to avoid CORS). A success/failure summary is displayed after sync.

🔗
Integration Note
The Jira Sync button is prominently featured in both the document toolbar (when FRD tab is active) and as a universal action in the top header toolbar for any document.
Section 13

E2E Test Automation — Playwright Script Generator

When the Test Cases tab is active and contains generated content, a dedicated "E2E Automation Suite" section appears below the test cases. This section generates executable Playwright test scripts.

13.1 "Generate Playwright Script" Button

Triggers a POST to /api/generate/tests with:

  • prototypeCode: HTML source from the generated Prototype (or Wireframe as fallback). If neither is available, a "no UI code" message is sent to guide the AI toward generic locators.
  • testCases: The full generated Test Cases document text

13.2 Validation Layer

The API endpoint runs a DR-RUN validation pass on the generated Playwright code before returning it. The response is JSON (not a stream) containing a code field with the validated TypeScript Playwright script.

13.3 Output Display

The generated script is displayed in a scrollable, syntax-highlighted code block with a "Copy" button overlay that appears on hover. The script uses proper Playwright locator strategies (by test ID, by role, by text) derived from the prototype HTML structure.

13.4 Requirements

  • Generated scripts must be valid TypeScript that can run with npx playwright test
  • Must include import { test, expect } from '@playwright/test'
  • Must cover all positive and negative test cases from the Test Cases document
  • Must handle async/await properly for all page interactions
Section 14

Infrastructure-as-Code (Terraform) Generator

When the SRD tab is active, a "Infrastructure-as-Code (Terraform)" section appears below the SRD content. This feature generates a deployable Terraform configuration derived from the technical requirements in the SRD.

14.1 "Generate IaC Manifest" Button

Triggers a POST to /api/generate/tests with prototypeCode: "GENERATE_IAC" and testCases set to the SRD content. The GENERATE_IAC sentinel value instructs the server to generate Terraform HCL instead of Playwright code.

14.2 Output

The generated Terraform plan is displayed in a monospace code block with a "Copy" button. The plan includes resource definitions for the infrastructure described in the SRD (e.g., cloud providers, databases, compute instances, networking, IAM roles).

14.3 Requirements

  • Generated HCL must be syntactically valid Terraform 1.x
  • Must use appropriate providers (AWS, GCP, Azure) based on technology stack mentioned in SRD
  • Must include variable blocks for sensitive values (passwords, API keys)
  • Must include a terraform required_providers block
Section 15

Collaboration & Real-time Editing

15.1 CollaborativeEditor Component

Text-based documents (BRD, FRD, PRD, SRD, Test Cases, Executive Pitch, Regulatory Advisor) are rendered in the CollaborativeEditor component (src/components/CollaborativeEditor.tsx) when the document has finished generating and is not in streaming mode.

The editor provides rich markdown editing capabilities and emits onUpdate(newMd) callbacks that immediately update the in-memory document state without requiring a re-generate. Changes persist in the database when the user uses "Save Session".

15.2 Edit Mode (Textarea Fallback)

For all document types, an "Edit" button in the toolbar toggles between view mode and a raw textarea editor. Clicking "Save" in edit mode replaces the document content with the edited version. This works for all document types including HTML Wireframes and Prototypes.

15.3 Version Tracking in Document Toolbar

The document toolbar displays the current version: "Baseline Version 1.0" for the initial generation, and "Version N.0 (Revised)" when requirements have evolved. The version number is derived from the length of the scopeHistory array.

15.4 Document Staleness Workflow

  1. Requirements change → Analyze API returns updated graph + delta
  2. BFS traversal identifies downstream stale document types
  3. Stale documents receive a visual warning indicator in the sidebar
  4. Cascading Impact panel shows affected documents + "Regenerate Stale Artifacts" button
  5. Clicking regenerate calls handleDocumentClick(docName, force=true)
Section 16

Authentication & Session Management

16.1 Authentication Provider

NextAuth.js v4 with Credentials provider (email + bcrypt-hashed password). JWT strategy is used — sessions are stored in HTTP-only cookies.

16.2 AuthModal Component

A modal-based sign-in/sign-up form (src/components/AuthModal.tsx) that appears when unauthenticated users click "Sign In to Save Work". The modal supports both sign-in and account creation flows without navigating away from the main workspace.

16.3 JWT Token Enrichment

On sign-in, the JWT callback enriches the token with organization context:

  • organizationId — the user's primary organization
  • orgName — display name shown in the sidebar
  • role — ADMIN | BA_LEAD | BA_ANALYST | VIEWER
  • plan — STARTER | PROFESSIONAL | ENTERPRISE
  • maxUsers — seat limit from the organization record
  • isOrgActive — whether the organization is active

16.4 Session Events

A signIn event logs a LOGIN audit entry to the organization's audit log via logAudit().

16.5 Rate Limiting

API routes are protected by a sliding-window rate limiter (src/lib/rate-limit.ts). PII detection (src/lib/pii.ts) scans inputs for sensitive data before passing to the AI. Permission checks (src/lib/permissions.ts) enforce role-based access control on all write operations.

Section 22

Security & Compliance

22.1 Authentication Security

  • Passwords hashed with bcrypt (salt rounds: 10+)
  • JWT tokens stored in HTTP-only, Secure, SameSite=Lax cookies
  • CSRF protection via NextAuth.js CSRF token on all state-mutating requests
  • No plaintext password storage in any log or response

22.2 Rate Limiting

A sliding-window rate limiter (src/lib/rate-limit.ts) limits AI API calls per IP per window. This prevents abuse of the Gemini API and protects against cost overruns.

22.3 PII Detection

The src/lib/pii.ts module scans user inputs for Personally Identifiable Information (names, emails, phone numbers, Aadhaar numbers, PAN numbers) before forwarding to the AI. Detected PII is either masked or the request is flagged for review depending on configuration.

22.4 Authorization

Role-based access control is enforced server-side via src/lib/permissions.ts. All API routes check the user's role from the JWT token before performing any write operation. Admin endpoints return 403 for non-admin roles.

22.5 Data Isolation

All database queries are scoped by userId and/or organizationId — there is no cross-tenant data leakage possible through the ORM layer. Prisma's type system ensures all queries include required scope filters.

22.6 Iframe Sandbox

The Prototype preview iframe runs with a restricted sandbox that prevents cookie access, top-level navigation, and access to the parent frame's DOM. External links within the prototype are intercepted by a load event handler.

22.7 Vercelignore & Environment Variables

Sensitive configuration (DATABASE_URL, NEXTAUTH_SECRET, GOOGLE_GENERATIVE_AI_API_KEY) is stored exclusively in Vercel environment variables and the .env.local file (gitignored). The .vercelignore file prevents unnecessary files from being deployed.
