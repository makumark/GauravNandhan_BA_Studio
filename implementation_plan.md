# Fix Generation Architecture\n\n## Goal\nStabilize the generation of all artifacts (FRD, Testcases, Logic Sandbox, Prototypes, etc.) by unifying the RAG architecture with the streaming pipeline.\n\n## Root Cause Analysis\n1. **Timeouts:** The new RAG endpoint (/api/generate-artifact) is synchronous. Heavy generations (Prototypes/UML) take over 30s, causing Node/Vercel timeout errors.\n2. **JSON Parsing Errors:** Logic Sandbox uses the streaming endpoint, which lacks the MANDATORY JSON SCHEMA prompt instructions. The LLM outputs markdown, breaking the JSON parser.\n3. **Inconsistent RAG:** FRD and Testcases use the streaming endpoint, which currently does not fetch GraphNode (RAG) data.\n\n## Proposed Changes\n\n### 1. Update /api/generate/stream/route.ts\n- Extract projectId from the request body.\n- Query prisma.graphNode.findMany({ where: { projectId } }) to fetch the RAG context.\n- If GraphNode is empty, fallback to fetching recent prisma.message context (the fix applied earlier).\n- Add the MANDATORY SCHEMA instruction to the prompt if isVisual is true (especially for Logic Sandbox and Prototypes).\n\n### 2. Update src/app/page.tsx\n- Pass projectId in the /api/generate/stream fetch payload.\n- Remove the conditional logic that routes visual artifacts to the non-streaming /api/generate-artifact endpoint.\n- Route ALL generation through /api/generate/stream so that everything benefits from both Streaming (no timeouts) and RAG (GraphNode context).\n\n## Verification\n- Generate a Prototype and verify it streams without timing out.\n- Generate a Logic Sandbox and verify it renders without JSON parsing errors.\n- Generate an FRD and verify it uses the RAG context.
